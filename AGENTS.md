# Agent 规范

## 沟通

- 代码注释：中文
- 代码术语：保留原文
- 用户交互：中文

## 规范

必须严格遵守 [openspec/project.md](openspec/project.md) 中的开发规范。

## 任务流程

必须严格遵守下面的流程：

1. 编码前先思考
   不要主观臆断，不要隐瞒疑问，清晰权衡取舍方案
   开始开发前：

- 明确列出你的各项假设，不确定就主动询问
- 存在多种理解方式时，全部罗列出来，不擅自私自选择
- 存在更简洁实现方案时主动说明，合理情况下敢于提出反对意见
- 内容存在模糊不清时立刻暂停，点明困惑点并及时沟通

2. 优先简洁实现
   用最少代码解决问题，不做多余推测拓展

- 不额外开发需求以外的功能
- 一次性使用的代码不做抽象封装
- 不擅自增加未要求的扩展性、可配置性
- 不对不可能出现的场景编写异常处理
- 如果你写了200行，发现能优化为50行时，务必重构简化

自我审视：资深工程师是否会认为这段代码过于复杂？如果是，就进一步简化。

3. 精准改动代码
   只修改必要内容，只清理自身产生的冗余
   修改存量代码时：

- 不擅自优化周边代码、注释、格式排版
- 不对运行正常的内容进行重构
- 贴合原有代码风格，即便你有更好写法也保持统一
- 发现无关无效代码，仅备注说明，不自行删除

自身改动产生废弃内容时：

- 删除本次修改后不再使用的导入、变量、函数
- 未经要求，不删除原本就存在的无效代码

检验标准：每一行修改内容，都能直接对应用户需求。

4. 围绕目标落地开发
   明确验收标准，反复校验直至符合要求
   将任务转化为可验证目标：
   新增校验逻辑 → 编写异常入参测试用例，确保用例通过
   修复问题漏洞 → 编写复现问题的测试用例，确保用例通过
   重构模块X → 保证重构前后所有测试用例均可正常通过

多步骤任务，简要梳理执行方案

1. 执行步骤 → 验证：核对结果
2. 执行步骤 → 验证：核对结果
3. 执行步骤 → 验证：核对结果

重复以上步骤，直到任务完成

## 禁止

- 主动 commit
- 引入不存在库
- 暴露密钥
- 不遵循 [openspec/project.md](openspec/project.md) 中的开发规范
